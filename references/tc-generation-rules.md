# 스펙 기반 TC 추출 가이드

> 스펙 문서를 읽고 TC(Test Case) 목록을 추출하기 위한 규칙.
> qa-review, web-qa, ui-qa 에이전트에서 참조.

## 작업 흐름

### Step 1: 스펙/위키 수집
- Confluence MCP(`confluence_get_page`)로 위키 페이지 읽기
- 하위 페이지, 링크된 문서도 함께 수집

### Step 2: 화면 구성 파악
TC 추출 전 화면 구성을 트리 형태로 정리:
```
{화면명}
├── {영역1} — 한 줄 설명
├── {영역2} — 한 줄 설명
│   ├── {하위 A} — 설명
│   └── {하위 B} — 설명
└── {영역3} — 설명
```

도출 우선순위:
1. Figma 디자인 확인 (최우선)
2. Chrome MCP로 위키 스크린샷
3. 스펙 텍스트에서 추출

### Step 3: 문서 성격 파악
| 성격 | 판단 근거 | TC 시 주의 |
|------|-----------|------------|
| 신규 기능 | 새 화면/기능 정의 | 모든 항목이 TC 대상 |
| 기존 개편 | as-is→to-be 비교 | "유지" = 존재 확인 TC 필요 |

### Step 4: 스펙 분석
각 구성 요소별 추출:
1. 동작 정의 (사용자 액션 → 시스템 반응)
2. 조건 분기 (상태별 다른 동작)
3. 경계값 (0개/1개/최대값)
4. 갱신/초기화 규칙
5. 로깅 요구사항

### Step 5: TC 목록 작성
### Step 6: 화면 구성 ↔ TC 크로스체크
### Step 7: UX 미확인 TC 디자인 검증

## TC 추출 기준

### 핵심 원칙
- **스펙에 명시된 동작만** TC로 만든다
- **하나의 TC = 하나의 검증 포인트**
- 포괄적 요약 문장은 TC 아님 (하위 TC가 대체)
- 레이아웃/UI 존재 여부는 TC 아님 (동작→반응만)

### TC 관점 체크리스트
- [ ] 초기 상태 / 기본값
- [ ] 사용자 인터랙션별 동작 (탭, 스와이프, 롱프레스 등)
- [ ] 상태 전환 (on↔off, 선택↔미선택)
- [ ] 조건 분기 (A이면 X, B이면 Y)
- [ ] 경계값 (0개, 1개, 최대값)
- [ ] 갱신/초기화 (화면 진입/이탈, 백그라운드)
- [ ] 다른 화면과의 연동
- [ ] 에러 케이스 (서버 통신 실패)

### 제외 대상
| 제외 | 예시 |
|------|------|
| UI 위치/배치 | "메모 영역은 하단에 노출" |
| UI 존재 여부 | "복사 아이콘이 제공된다" |
| 정적 안내문구 | "최대 3개까지 설정 가능합니다" |
| 고정 표시값 | "참석 영문명은 'Going'" |

## 출력 포맷

```markdown
# {기능명} TC 목록

## 화면 구성
(트리 구조)

## {카테고리} ({N}개)
> @Suite("{카테고리 테스트}")

| # | @Test |
|---|-------|
| 1 | "{TC 제목}" |

## [추론] 에러 케이스 ({N}개)
| # | 추론 근거 | @Test |
|---|-----------|-------|

## [기획자 확인 필요] UX 미확인 TC ({N}개)
| # | 카테고리 | 스펙 원문 | 확인 필요 사항 |
|---|---------|-----------|---------------|

## [참고] 화면 외 정책 ({N}개)
| # | 스펙 내용 | 검증 화면 |
|---|-----------|-----------|

**총 TC 수: 스펙 {N}개 + 추론 에러 {M}개 = {총합}개**
```

## 상세 규칙

전체 상세 규칙은 원본 문서 참조:
- 입력/설정 화면 TC 분리 원칙 (입력 → 상태변경 → 저장)
- UI 트리거 추론 원칙 (3단계 분기)
- 연쇄 동작 분리
- 화면 외 정책 분리
- 에러 케이스 추론
